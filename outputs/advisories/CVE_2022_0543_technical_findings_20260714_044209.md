> **Scope Note:** This vulnerability is **Debian/Ubuntu packaging-specific**. Redis compiled from upstream source is not affected. The root cause is that Debian's packaging exposes the Lua `package` library as a global within the Redis Lua sandbox, allowing sandbox escape via `package.loadlib()`.

## Attack Vector

### Exploitation Mechanism

Redis embeds a Lua scripting engine accessible via the `EVAL` command. In correctly hardened Redis builds, the Lua `package` and `io` libraries are stripped from the sandbox. In Debian-packaged Redis, the `package` library is inadvertently retained as a global — likely due to dynamic linking of the Lua shared library (`liblua5.1.so`) and the way Debian patches initialise the interpreter.

An attacker exploits this by calling `EVAL` with a crafted Lua script that invokes `package.loadlib()` to load a native shared library from the host filesystem, gaining arbitrary code execution in the context of the Redis process (commonly running as `redis` user, but in misconfigured environments as `root`).

**Minimal proof-of-concept pattern (sanitised — no weaponised payload):**
EVAL "local io_l = package.loadlib('/usr/lib/x86_64-linux-gnu/liblua5.1.so.0', 'luaopen_io'); ..." 0

### Network Path and Authentication Requirements

| Component | Detail |
|---|---|
| **Attack Vector** | Network (AV:N) — Redis TCP port, default `6379/TCP` |
| **Complexity** | Low (AC:L) — no race condition, reliable exploitation |
| **Privileges Required** | None (PR:N) — **if Redis has no authentication configured (no `requirepass` / ACLs)** |
| **Privileges Required (auth)** | Low — any valid Redis credential sufficient; `EVAL` is not restricted by default ACL |
| **User Interaction** | None (UI:N) |
| **Scope** | Changed (S:C) — sandbox escape crosses the Lua VM boundary |
| **Impact** | Confidentiality: High, Integrity: High, Availability: High |

**Critical condition:** Redis instances exposed to the internet without `requirepass` or bound only to `0.0.0.0` without firewall controls are trivially reachable. CISA KEV confirmation indicates active exploitation against exactly these configurations.

**Authentication bypass note:** In many observed exploitation campaigns, attackers use prior credential theft, Redis CONFIG SET to write SSH keys, or simply unauthenticated access. Even with authentication, any account with `EVAL` access (all users by default in Redis ACL unless restricted) is sufficient.

## Observable Behaviour

### Network Indicators

| Indicator | Detail |
|---|---|
| **Destination Port** | `6379/TCP` (default Redis); also `6380/TCP` (TLS Redis) |
| **Protocol** | Redis Serialization Protocol (RESP) — plaintext, line-oriented |
| **Source** | Scanning activity from threat actor infrastructure; look for high-rate `6379/TCP` SYN sweeps |

**RESP wire pattern for malicious EVAL command:**
*3\r\n$4\r\nEVAL\r\n$<length>\r\n<lua_payload_containing_package.loadlib>\r\n$1\r\n0\r\n

Specifically, look for RESP streams containing the ASCII strings:
- `package.loadlib` within an `EVAL` argument
- `liblua` within an `EVAL` argument
- `luaopen_io` or `luaopen_os` within an `EVAL` argument
- `io.popen` within an `EVAL` argument (post-sandbox-escape execution)

### Endpoint / Process Telemetry

Following successful exploitation, the Redis process (`redis-server`) will spawn child processes. Key process chain indicators:

redis-server (pid: XXXX)
  └─ sh -c <attacker command>          ← suspicious: redis-server should not spawn shells
       └─ wget / curl <C2 URL>         ← download of secondary payload
       └─ chmod +x / ./malware

**Specific endpoint indicators to hunt:**

| Indicator Type | Value |
|---|---|
| **Parent process** | `redis-server` spawning `sh`, `bash`, `python`, `perl`, `curl`, `wget` |
| **File writes by redis-server** | Writes to `/etc/cron.d/`, `/root/.ssh/authorized_keys`, `/tmp/*.sh`, `/var/spool/cron/` |
| **Network connections from redis-server** | Outbound connections on non-Redis ports (e.g., `80/TCP`, `443/TCP`, `4444/TCP`) initiated by `redis-server` PID |
| **Loaded libraries** | `liblua5.1.so` loaded followed by `libio.so` or similar via `dlopen()` syscall from within `redis-server` |

**Known campaign TTPs (post-exploitation, as documented in threat intel):**
- Deployment of XMRig cryptocurrency miners
- Persistence via crontab modification
- SSH key injection into `~redis/.ssh/authorized_keys` or `/root/.ssh/authorized_keys`
- Reverse shell establishment to C2 infrastructure
