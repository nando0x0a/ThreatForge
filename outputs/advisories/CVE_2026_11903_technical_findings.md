## Attack Vector

### Exploitation Mechanism

This is a **stored (persistent) XSS** vulnerability [3] residing in MOVEit Transfer's Ad Hoc Transfer module. Unlike reflected XSS, stored XSS payloads are written to the application's backend storage and served to subsequent victims who load the affected page — meaning a single attacker action can affect multiple users over time.

**Exploitation chain:**

1. **Attacker** submits a crafted input (e.g., a malicious filename, message field, or recipient field within the Ad Hoc module) containing unsanitized script content.
2. The application fails to neutralize the input before persisting it [1][3], storing the raw payload server-side.
3. A **victim user** (authenticated MOVEit Transfer user) navigates to the affected page — e.g., viewing a transfer notification, file listing, or Ad Hoc package detail.
4. The victim's browser renders the stored payload, executing attacker-controlled JavaScript in the victim's authenticated session context.

### Required Conditions

| Condition | Detail |
|---|---|
| **Network access** | The attacker must be able to reach the MOVEit Transfer web interface (typically TCP/443 or TCP/80) |
| **Attacker authentication** | Likely requires at least a low-privilege MOVEit account to submit content through the Ad Hoc module (specific authentication requirement not confirmed in available advisories — analyst verification required) |
| **Victim interaction** | **Required** — a victim user must browse to the page rendering the stored payload [3] |
| **Attack complexity** | Low — once stored, payload delivery is passive |

### CVSS Context

The score of **8.0 (HIGH)** [1] reflects meaningful impact despite the user-interaction requirement. The stored nature elevates risk above typical reflected XSS: the payload persists until removed, can target high-privileged administrators, and can facilitate session hijacking, credential theft, or lateral movement within the MOVEit web application context.

## Observable Behaviour

### Attack Phase — Payload Injection

An attacker writing a stored XSS payload via the Ad Hoc module would generate HTTP POST requests to MOVEit Transfer endpoints. Specific URI paths for the Ad Hoc module are not confirmed in the available advisory content [2]; the following patterns are representative and must be validated against the deployed application.

**Indicative HTTP request pattern (POST — payload injection):**
POST /moveitisapi/moveitisapi.dll?action=adhoc HTTP/1.1
Host: <moveit-transfer-host>
Content-Type: application/x-www-form-urlencoded
Cookie: ASP.NET_SessionId=<session>

...fieldname=<script>/* attacker payload */</script>...

**Common XSS payload patterns to look for in request bodies and logged parameters:**
<script>
<img src=x onerror=
<svg onload=
javascript:
&#x3C;script&#x3E;   (HTML-encoded variants)
%3Cscript%3E        (URL-encoded variants)

### Attack Phase — Payload Execution (Victim Side)

When a victim loads the affected page, the stored script executes in their browser. Observable artifacts:

- **Outbound HTTP/S requests** from victim workstation to attacker-controlled infrastructure (e.g., cookie exfiltration via `document.cookie` sent to external host)
- **Unexpected JavaScript-initiated network connections** originating from the browser process on endpoints running MOVEit Transfer web clients
- **DOM-based anomalies** visible in browser developer tools / endpoint EDR browser telemetry

### Log Sources to Monitor

| Log Source | What to Look For |
|---|---|
| **MOVEit Transfer application logs** | POST requests to Ad Hoc module endpoints containing `<script`, `onerror=`, `onload=`, `javascript:` in parameter values |
| **Web Application Firewall logs** | XSS signature alerts against MOVEit Transfer FQDN |
| **Proxy / DNS logs** | Victim workstations making unexpected outbound connections to unknown hosts shortly after browsing MOVEit |
| **SIEM** | Correlate MOVEit user session activity with subsequent anomalous outbound connections |

> **Note:** Specific MOVEit Transfer log file paths and formats should be confirmed from Progress documentation [2]. Default log location on Windows installations is typically `C:\MOVEitTransfer\Logs\` but should be verified.

## Detection Coverage

> Detection rules below are **proposed drafts for analyst review**. Tune thresholds, paths, and signatures against your specific environment before deployment. Test in alert-only mode before enforcing.

### Suricata Rule (Draft)

alert http $EXTERNAL_NET any -> $HTTP_SERVERS any (
    msg:"ThreatForge DRAFT - CVE-2026-11903 MOVEit Transfer Ad Hoc Stored XSS Payload Attempt";
    flow:established,to_server;
    http.method; content:"POST";
    http.uri; content:"/moveitisapi/"; nocase;
    http.request_body;
    pcre:"/(<script[\s>]|javascript\s*:|on(error|load|click|mouseover)\s*=|<svg[\s>]|<img[^>]+onerror\s*=)/i";
    classtype:web-application-attack;
    sid:20261190301;
    rev:1;
    metadata:cve CVE-2026-11903, created_at 2026, affected_product MOVEit_Transfer;
)

**Analyst notes on this rule:**
- URI path `/moveitisapi/` is representative; confirm actual Ad Hoc module endpoint path from the Progress advisory [2] or application traffic baseline.
- The PCRE targets common XSS injection patterns; it will generate false positives if users legitimately submit HTML content. Whitelist known-safe senders if needed.
- Encoded variants (`%3Cscript%3E`, `&#x3c;`) may evade this rule — consider adding a companion rule with `urilen` and base64/HTML-decode normalization enabled in your IDS configuration.
