**Summary:** CVE-2021-42013 is a path traversal vulnerability in Apache HTTP Server versions 2.4.49 and 2.4.50 that bypasses the incomplete fix introduced for CVE-2021-41773. An unauthenticated remote attacker can traverse outside the document root using encoded path sequences. When mod_cgi or mod_cgid is enabled, traversal primitives chain into unauthenticated remote code execution (RCE). CISA has confirmed active in-the-wild exploitation.

### Exploitation Mechanism

**Phase 1 — Path Traversal:**
Apache 2.4.49 introduced normalisation handling for URL paths, which CVE-2021-41773 partially patched. CVE-2021-42013 demonstrates that double-encoded or alternative percent-encoded sequences bypass that normalisation:

- **Single-encoded (blocked by 2.4.49 patch):** `%2e%2e/` → `../`
- **Double-encoded (exploits 2.4.42013):** `%%32%65` → `%2e` → `.`
- **Alternative encoding:** `.%2e/` or `%2e./`

This allows an attacker to escape from a configured `Alias` or `DocumentRoot` boundary. For example, a request such as:

GET /cgi-bin/.%%32%65/.%%32%65/.%%32%65/.%%32%65/etc/passwd HTTP/1.1
Host: <target>

...causes Apache to serve `/etc/passwd` if `require all denied` is not explicitly configured on the target directory.

**Required Conditions (either satisfies partial or full exploitation):**

| Condition | Result |
|---|---|
| Alias/ScriptAlias directive configured (e.g., `/cgi-bin/`) AND `require all denied` absent on filesystem directories | Path traversal → arbitrary file read |
| `mod_cgi` or `mod_cgid` enabled on the ScriptAlias path | Full unauthenticated RCE via POST request with shell commands in body |

**Phase 2 — RCE via mod_cgi:**
If CGI is enabled under a traversable alias (common default in many distributions), the attacker issues a POST request, with shell command execution delivered in the HTTP body:

POST /cgi-bin/.%%32%65/.%%32%65/.%%32%65/.%%32%65/bin/sh HTTP/1.1
Host: <target>
Content-Type: application/x-www-form-urlencoded

echo Content-Type: text/plain; echo; id; uname -a; cat /etc/passwd

The Apache process executes `/bin/sh` directly with the POST body piped as stdin, returning command output in the HTTP response body. The process runs under the `www-data`, `apache`, or `daemon` user context depending on server configuration.

**Authentication:** None required. No session tokens, credentials, or pre-existing access needed.

**HTTP method:** `POST` is definitive for RCE; `GET` may indicate reconnaissance/file read.

**HTTP response codes to monitor:**
- `200 OK` on a traversal URI → successful exploitation
- `403 Forbidden` → server configured correctly, attack blocked
- `400 Bad Request` → some WAFs/proxy-normalising infrastructure blocking
- `500 Internal Server Error` → potential CGI misconfiguration or partial hit
