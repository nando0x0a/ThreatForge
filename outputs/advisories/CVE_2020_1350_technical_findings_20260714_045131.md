## Attack Vector

### Exploitation Mechanism

CVE-2020-1350 is a **heap-based buffer overflow** in the Windows DNS Server service (`dns.exe`) triggered during the parsing of **SIG (Signature) resource records** in DNS responses. The overflow occurs because the server incorrectly validates the length of SIG record data, allowing an attacker-controlled value to cause a write beyond the bounds of the allocated heap buffer.

**Wormability:** The vulnerability is classified as wormable (CVSS Scope: Changed). A compromised DNS server can be weaponised to laterally attack other Windows DNS servers without further attacker interaction, similar in risk profile to EternalBlue/MS17-010.

### Network Path

Attacker (external or internal) 
    │
    └─► Windows DNS Server (UDP/53 or TCP/53)
            │
            └─► dns.exe parses malicious SIG RR response
                    │
                    └─► Heap overflow → RCE as SYSTEM (NT AUTHORITY\SYSTEM)

**Two primary attack paths exist:**

1. **Direct attack (UDP/TCP port 53):** An attacker sends a crafted DNS query or response containing a malformed SIG resource record directly to an exposed DNS server. This is possible if port 53 is reachable from an adversary-controlled host.

2. **Forwarded/recursive resolution attack:** The attacker registers a malicious domain and configures its authoritative nameserver to return a malformed SIG record. When the target Windows DNS server performs recursive resolution for that domain (on behalf of any internal client), it receives and parses the malformed response — triggering exploitation without direct inbound access to port 53.

## Observable Behaviour

### Network-Layer Indicators

**Protocol:** DNS over UDP port 53 (initial contact) or TCP port 53 (large response fallback — notably, SIG records large enough to trigger the overflow force TCP fallback).

**Key network indicator — TCP fallback for large DNS responses:**
DNS response size > 65,535 bytes (or SIG RDATA length field value ≥ 0xFFFF)
Transport: TCP/53 (forced fallback from UDP due to response size)
Resource Record Type: SIG (RRTYPE 24) or KEY (RRTYPE 25)

**Malformed SIG record structure (conceptual, not exploit code):**
DNS Answer Section:
  Name:    <attacker-controlled domain>
  Type:    SIG (24)
  Class:   IN (1)
  TTL:     <arbitrary>
  RDLENGTH: <value near or at 0xFFFF — anomalously large>
  RDATA:   <oversized / malformed SIG record data>

A legitimate SIG RDATA length will be proportional to the signature algorithm and key material involved. Values approaching or reporting `0xFFFF` (65535 bytes) in the RDLENGTH field are anomalous and should be treated as suspicious.

**DNS query patterns to monitor:**
- Queries from the Windows DNS server to external domains that subsequently return TCP-transported responses
- DNS responses inbound on TCP/53 from external resolvers with unusually large answer sections
- Repeated SERVFAIL or server-side DNS crashes correlated with resolution of a specific external domain

### Endpoint / Process Telemetry

Faulting module: dns.exe or ntdll.dll
Exception code: 0xC0000005 (Access Violation) or 0xC0000374 (Heap Corruption)

**Post-exploitation process chain (if exploitation achieves code execution):**
dns.exe (SYSTEM)
  └─► cmd.exe or powershell.exe          ← command execution
        └─► net.exe / whoami.exe         ← reconnaissance
        └─► certutil.exe / bitsadmin.exe ← payload retrieval (common LOLBins)

**Sysmon Event IDs to correlate:**
Event ID 1  (Process Create): dns.exe spawning cmd.exe, powershell.exe, wscript.exe
Event ID 3  (Network Connect): Unexpected outbound connections from dns.exe
Event ID 7  (Image Load): Unusual DLL loads into dns.exe process space
Event ID 10 (Process Access): dns.exe being accessed by unexpected parent

- Crash offset in `dns.exe` or heap corruption in `ntdll.dll` heap management routines
- Stack traces referencing DNS SIG or KEY record parsing functions
