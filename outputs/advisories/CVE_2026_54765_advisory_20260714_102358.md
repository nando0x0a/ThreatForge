# Security Advisory: Traefik Kubernetes Gateway Access Control Vulnerability

**Reference:** CVE-2026-54765
**Date Issued:** [Draft — Pending Analyst Review]
**Priority Tier:** MONITOR
**Severity:** HIGH (NVD) / MEDIUM (CNA) — see Severity Note below

---

## Executive Summary

A security vulnerability has been identified in Traefik, a widely used network traffic routing component commonly deployed in Kubernetes environments. The flaw allows an individual with limited, legitimate access to the system — specifically, the ability to create certain network routing rules — to bypass intended boundaries between isolated application environments and apply unauthorised configurations that affect services they should not be able to reach. This represents a breakdown in one of the fundamental access controls designed to keep different application teams or tenants separated. While a patch is available, the internal risk assessment places this vulnerability in the **MONITOR** tier, meaning it does not require emergency action but should be scheduled for remediation in a structured, timely manner.

> **⚠️ Severity Note for Analyst Review:** Two authoritative sources disagree on the severity of this vulnerability. The National Vulnerability Database rates it **8.5 (HIGH)** [1], while the CVE Programme's own publishing authority rates it **6.3 (MEDIUM)** using the newer CVSS v4.0 scoring standard [4]. This discrepancy is material and should be reviewed by a security analyst before communicating severity to stakeholders. The higher score may reflect worst-case exploitation potential; the lower score may better reflect real-world exploitability constraints. This advisory uses HIGH as a conservative default pending analyst determination.

---

## Business Impact

If this vulnerability is not addressed, the following business risks apply:

- **Unauthorised access to internal services:** An attacker — or a malicious insider — who can create routing rules within the Kubernetes environment could redirect or intercept traffic destined for services in other, nominally separate parts of the infrastructure. This could expose sensitive internal APIs, data services, or administrative endpoints to parties who should have no access.

- **Multi-tenant environment risk:** Organisations running multiple application teams, customers, or business units on shared Kubernetes infrastructure face elevated risk. The separation between these groups (a key security and compliance control) could be undermined.

- **Compliance and regulatory exposure:** Depending on the nature of the services exposed, a successful exploitation could result in unauthorised access to regulated data (e.g., personal data under GDPR, financial data under PCI-DSS), triggering notification obligations or audit findings.

- **Limited blast radius with access prerequisite:** It is important to note that an attacker must already possess the ability to create accepted network routing rules within the environment to exploit this flaw. This reduces — but does not eliminate — the risk, particularly in environments where developer or operator access is broadly granted.

## Affected Systems

The following product and version range is affected:

| Product | Affected Versions | Status |
|---|---|---|
| **Traefik** (Kubernetes Gateway API deployments) | Version 3.7.0 up to, but not including, 3.7.6 | Patched in v3.7.6 [2][3] |

> **Note:** This vulnerability specifically affects Traefik deployments using the **Kubernetes Gateway API provider**. Deployments not using this feature may not be exposed. IT teams should confirm which configuration is in use.

---

## Recommended Action

Management is asked to **approve and communicate** the following actions to the relevant infrastructure and platform engineering teams:

1. **Identify exposure within 5 business days:** Direct the platform or DevOps team to confirm whether Traefik is deployed in the affected version range and whether the Kubernetes Gateway API provider is in use.

2. **Schedule upgrade to Traefik v3.7.6 or later** [2] within the standard change management cycle — targeting completion within **30 days**. No emergency change window is required at this time given the MONITOR priority tier.

3. **Review access controls in the interim:** As a precautionary measure, ask the security or platform team to confirm that the ability to create Kubernetes HTTPRoute resources is restricted to trusted, vetted personnel only. Tightening this access reduces exploitation risk while the patch is being scheduled.

4. **Analyst severity review:** Given the disagreement between severity ratings from two authoritative sources [1][4], a security analyst should confirm the final severity assessment before any broader stakeholder communications are issued.
