## Observable Behaviour Patterns

*Given the nature of this vulnerability (CORS misconfiguration enabling cross-origin code execution), specific file/network IoCs are unlikely to be present in logs. Analysts should monitor for the following behavioural patterns instead:*

1. **Cross-origin requests to `/api/v1/functions`**
   Confidence: HIGH
   Look for HTTP requests to the functions API endpoint where the `Origin` header is set to an external, non-trusted domain. A permissive `allow_origins=*` configuration [3] means these requests will not be blocked by the server.

2. **Unexpected function creation or modification by admin accounts**
   Confidence: HIGH
   Since exploitation requires an authenticated admin session [3], forensic indicators may appear as unexpected new or modified functions created in the open-webui admin panel without corresponding legitimate admin activity.

3. **Admin session activity coinciding with external page visits**
   Confidence: MEDIUM
   Correlate admin account activity logs against time windows when admin users may have visited external/unknown websites. Exploit chain requires admin to visit an attacker-controlled page [3].

4. **Anomalous outbound connections from the open-webui server**
   Confidence: MEDIUM
   Post-exploitation RCE may result in outbound connections (reverse shells, data exfiltration). Monitor for unexpected outbound traffic from the host running open-webui, particularly on non-standard ports.

5. **Public PoC activity scanning for the endpoint**
   Confidence: MEDIUM
   With at least one public PoC known to exist [5], opportunistic scanning for `/api/v1/functions` may appear in web server access logs from external IPs, even without a valid session.
