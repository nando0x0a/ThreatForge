## URI Path Indicators
*Confidence: MEDIUM — derived from vulnerability mechanics, not confirmed exploitation traffic.*

URI: /api/git/*/content  # Potential path for reading excluded directory contents via case-insensitive bypass
URI: /api/git/*/show     # Git file content retrieval endpoint — may be abused to read excluded paths
URI: /api/git/*/diff     # Diff endpoints may expose excluded file content via case variation in path arguments

> **Note:** The vulnerability resides in `GitHandler.prepare()` in JupyterLab Git < 0.54.0 [3], where `excluded_paths` enforcement uses `fnmatch.fnmatchcase()` [1][4]. On case-insensitive filesystems (e.g., macOS, Windows NTFS), an authenticated user can bypass path exclusions by requesting excluded directories with altered casing (e.g., `.ENV` instead of `.env`, `SECRETS` instead of `secrets`). URI paths above represent plausible access vectors but have **not** been confirmed from exploitation telemetry.

## Observable Behaviour Patterns
*In the absence of confirmed IoCs, the following behavioural patterns should be monitored:*

1. **Authenticated requests to Git API endpoints referencing paths that match known excluded directories using case variations** — e.g., accessing `.ENV`, `.Env`, `Config`, `CREDENTIALS` where lowercase equivalents are in `excluded_paths`.
2. **Repeated or scripted enumeration of Git-related API endpoints** from a single authenticated session, particularly targeting non-standard path casings.
3. **Access to sensitive file types** (`.env`, `secrets/`, `credentials/`, `config/`) via JupyterLab Git API endpoints on servers running on case-insensitive filesystems (macOS, Windows).
4. **Upgrade absence:** Instances still running jupyterlab-git < 0.54.0 [3] after patch availability should be flagged as exposure risk.

## Notes

- **No public PoC is known** as of the date of this report; exploitation would require an authenticated attacker with access to the JupyterLab instance.
- **CVSS 7.1 (HIGH)** [1] reflects meaningful risk due to potential exposure of secrets or configuration files, though authentication requirement limits the attack surface.

- **Filesystem dependency is critical:** Exploitation is only possible on case-insensitive filesystems [1]. Linux deployments (case-sensitive ext4 etc.) are not vulnerable to this specific bypass.
- **Patch reference:** Fix is present in commit `460035275b5963dc96e364e60ba6a73717fbd033` [2] and release v0.54.0 [3]. Recommend upgrading immediately.
