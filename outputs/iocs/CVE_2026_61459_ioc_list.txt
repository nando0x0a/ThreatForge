## Observable Behaviour Patterns

Given the absence of specific IoCs in the provided context [1][2][3][4], the following **observable behaviour patterns** should be used as detection anchors:

### Pattern 1 ŌĆö Argument Injection via resourceType / name Parameters
**Confidence: HIGH**
Exploitation requires an attacker to supply `resourceType` or `name` parameters beginning with leading dashes (e.g., `--server=https://attacker.example.com`) to bypass the `assertNoDangerousFlags` check [1][2][3].

- **Observe:** MCP tool call payloads where `resourceType` or `name` field values begin with `-` or `--`
- **Observe:** Any value containing `--server=`, `--kubeconfig=`, `--certificate-authority=`, `--insecure-skip-tls-verify` injected via these fields

### Pattern 2 ŌĆö Unexpected kubectl `--server` Flag Invocation
**Confidence: HIGH**
Successful exploitation redirects kubectl commands to an attacker-controlled API server [1][2].

- **Observe:** Process execution logs showing `kubectl` invoked with a `--server` argument pointing to an IP or domain not matching your legitimate cluster endpoint
- **Observe:** kubectl process trees spawned by the MCP server process with unfamiliar `--server` values

### Pattern 3 ŌĆö Bearer Token Exfiltration via Outbound TLS
**Confidence: HIGH**
The vulnerability allows bearer token leakage through the redirected kubectl connection [1][3].

- **Observe:** Outbound TLS connections from the host running `mcp-server-kubernetes` (or its container) to unexpected external IP:6443 or IP:443 targets
- **Observe:** Kubernetes service account token files accessed immediately prior to anomalous outbound connections

### Pattern 4 ŌĆö Vulnerable Version in Use
**Confidence: HIGH**
Any deployment of `mcp-server-kubernetes` **< 3.9.0** is potentially vulnerable [1][2].

- **Observe:** Package manifests, container images, or node_modules containing `mcp-server-kubernetes` at versions below `3.9.0`
- **Artifact type:** `package.json` dependency entries, container image labels, npm lock files
