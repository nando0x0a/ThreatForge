## Observable Behaviour Patterns
*(Listed in place of specific IoCs due to absence of provided OSINT — CONFIDENCE: MEDIUM)*

**1. Redis Protocol Abuse — Lua Sandbox Escape Payload Pattern**
- BEHAVIOUR: Redis `EVAL` command submitted with Lua code referencing the
  `package` or `require` global — specifically `require('io')`, `require('os')`,
  or `package.loadlib(...)` — which are normally inaccessible within the Redis
  Lua sandbox but exposed on vulnerable Debian/Ubuntu builds due to dynamic
  linking of the Lua library (`liblua5.1.so`).
- PROTOCOL: TCP/6379 (default Redis port; also 6380 for TLS or non-standard configs)
- CONFIDENCE: HIGH (this is the documented vulnerability mechanism)

**Example observable command pattern (detection-only — not exploit code):**
EVAL "local io = require('io'); ..." 0
EVAL "local os = require('os'); ..." 0
EVAL "package.loadlib(...)" 0
> Analysts: Alert on `EVAL` commands containing `require` or `package.loadlib`
> in Redis command logs or network captures. Legitimate Redis Lua scripts should
> NOT reference these globals.

---

**2. Process Spawn from Redis Parent**
- BEHAVIOUR: Child processes (e.g., `sh`, `bash`, `curl`, `wget`, `python`,
  `perl`) spawned with `redis-server` or `redis` as the parent process.
- DATA SOURCE: EDR process tree, auditd, syslog
- CONFIDENCE: HIGH (indicative of successful RCE via sandbox escape)

---

**3. Outbound Network Connections from Redis Process**
- BEHAVIOUR: `redis-server` process initiating outbound TCP connections,
  particularly to non-local, non-RFC1918 addresses on ports 80, 443, 4444,
  or other common C2/download ports.
- DATA SOURCE: Network flow logs, EDR, `/proc/[pid]/net/tcp`
- CONFIDENCE: HIGH (Redis should not initiate outbound connections in normal operation)

---

**4. File Writes by Redis Process to Unexpected Paths**
- BEHAVIOUR: Redis process writing files outside of its configured `dir`
  (commonly `/var/lib/redis`) — e.g., writing to `/tmp/`, `/var/tmp/`,
  `/etc/cron.d/`, `/root/.ssh/authorized_keys`.
- DATA SOURCE: auditd (syscall `open`/`write` with `redis` UID), EDR file events
- CONFIDENCE: HIGH (persistence mechanism post-exploitation)

---

**5. Cron/Authorized_keys Persistence**
- BEHAVIOUR: New entries in `/etc/cron.*` or `/var/spool/cron/` or
  `/root/.ssh/authorized_keys` created by or attributable to the Redis
  process user — consistent with documented Lua RCE post-exploitation
  patterns for this CVE.
- CONFIDENCE: MEDIUM (common post-exploitation pattern; not CVE-specific alone)

---

**6. Exposure of Redis on Public Interface**
- BEHAVIOUR: `redis-server` bound to `0.0.0.0` or a public IP without
  authentication (`requirepass` not set) — a prerequisite for unauthenticated
  exploitation in most observed campaigns.
- DATA SOURCE: Shodan queries (`port:6379 product:Redis`), internal network
  scanning, Redis `CONFIG GET bind` / `CONFIG GET requirepass`
- CONFIDENCE: HIGH (precondition indicator)

## Notes

1. **Scope:** CVE-2022-0543 affects **Debian and Ubuntu Redis packages only**,
   due to how those distributions compile Redis with a shared Lua library,
   inadvertently exposing `package` globals. Upstream Redis (compiled from
   source) and non-Debian distributions are **not affected**.

2. **Affected versions:** Redis packages on Debian Buster, Bullseye, and Ubuntu
   focal/bionic prior to patched package versions. Analysts should verify
   installed package versions via `dpkg -l redis-server`.
