## Observable Behaviour Patterns
*Because no PoC is public and no in-the-wild exploitation is confirmed, observable patterns are more actionable than specific IoCs at this time [1][2].*

**1. CSRF Token Absence on GET Requests (Confidence: HIGH)**
- Splunk Web GET requests to Deployment Server endpoints that are **missing CSRF tokens** where tokens are expected. This is the root condition enabling exploitation [2].
- Detection target: Splunk Web access logs showing GET requests to deployment server URLs without a valid `splunk_form_key` or equivalent token parameter.

**2. Unexpected SPL Searches Executed as `splunk-system-user` (Confidence: HIGH)**
- SPL searches initiated under the `splunk-system-user` identity that were not scheduled or triggered through expected automation [2].
- Detection target: Splunk `_audit` index, `search` sourcetype — filter for `user=splunk-system-user` AND unexpected or anomalous search strings.

**3. Cross-Origin / Referrer Anomalies on Deployment Server Endpoints (Confidence: MEDIUM)**
- HTTP requests to Deployment Server endpoints with an external or unexpected `Referer` / `Origin` header, consistent with a cross-site request sourced from an attacker-controlled page [2].
- Detection target: Splunk Web access logs, `nginx`/`httpd` proxy logs.

**4. Victim User with `list_deployment_server` Role Initiating Unusual Browser Requests (Confidence: MEDIUM)**
- Audit events showing a user with the `list_deployment_server` role performing searches or triggering deployment server activity outside of normal operational hours or from an unusual IP/device — consistent with a phishing-initiated session [2].

**5. Stored XSS Payload in Deployment Server Configuration (Confidence: MEDIUM)**
- Review of Deployment Server app configurations, serverclass configurations, or any user-controlled input fields rendered in Splunk Web for unexpected script tags, JavaScript URI schemes, or encoded payloads [2].
- Suggested search (for analyst adaptation):
  
  index=_internal OR index=main sourcetype=splunk_web_access uri=*deployment*
  | search referer!=https://your-splunk-instance*
