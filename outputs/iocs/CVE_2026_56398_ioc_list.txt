## URI Path Indicators

**Confidence: MEDIUM** — Derived from advisory description [2][3]; specific
path segments inferred from vulnerability class (OAuth profile image endpoint).
Analyst should confirm exact path against their Open WebUI deployment version.

URI: /oauth/                         # OAuth authentication flow entry point
URI: /api/v1/auths/signin            # Likely OAuth callback/session handler
URI: /api/v1/users/*/profile-image   # Profile image endpoint (wildcard user ID)
URI: /static/                        # Potential SVG payload delivery via static assets

*The vulnerability resides in the OAuth authentication flow profile image
endpoint where MIME type validation is absent or insufficient, allowing
SVG files containing `<script>` tags or JavaScript URIs (e.g.,
`data:image/svg+xml;base64,...`) to be stored and served [2][3].*

## Behavioural Observables

Given the stored XSS via SVG Data URI nature of this vulnerability [3], the
following observable patterns should be prioritised for detection in lieu of
hard IoCs:

**1. Malicious OAuth `picture` Claim (HIGH confidence pattern)**
- OAuth JWT/OIDC `picture` claim containing a `data:image/svg+xml` URI
  (base64-encoded or raw) rather than an `https://` image URL [3]
- Specifically: `picture` claim values beginning with:
  - `data:image/svg+xml`
  - `data:text/html`
  - `javascript:`

**2. SVG Content with Active Script Content**
- Profile image uploads or OAuth-supplied image URIs containing:
  - `<script>` tags within SVG markup
  - `onload=` / `onerror=` event handlers in SVG elements
  - `<foreignObject>` with embedded HTML
  - `href="javascript:..."` within SVG `<a>` or `<use>` elements

**3. Suspicious Token Exfiltration Patterns**
- Outbound HTTP requests from authenticated browser sessions to
  unexpected third-party domains shortly after profile image rendering
- POST/GET requests carrying `localStorage` or `sessionStorage` token
  values to non-application endpoints (indicative of XSS token theft [2])

**4. Anomalous OAuth Login Activity**
- New or previously unseen OAuth accounts (especially those with
  non-standard `picture` claim formats) authenticating into the system
  on instances running Open WebUI < 0.9.5 [1][2][3]
