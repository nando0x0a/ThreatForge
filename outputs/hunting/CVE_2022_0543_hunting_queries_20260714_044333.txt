## CrowdStrike Event Search

### Query 1 — Outbound Network Connections Originating from Redis Process
*Targets: C2 beaconing or lateral movement initiated by redis-server after successful Lua sandbox escape and RCE.*

event_simpleName IN ("NetworkConnectIP4", "NetworkConnectIP6")
  ImageFileName IN ("*redis-server*", "*/redis/*")
| eval RemotePort=RemotePort
| where RemotePort != 6379 AND RemotePort != 16379
| stats count, values(RemoteIP) as RemoteIPs, values(RemotePort) as RemotePorts,
        dc(RemoteIP) as UniqueRemoteIPs
  by ComputerName, UserName, ImageFileName, LocalIP
| where count > 1 OR UniqueRemoteIPs > 1
| sort -count
> **Analyst Note:** Redis should almost never initiate outbound connections to arbitrary remote IPs. Any hit here warrants immediate investigation. Port 6379 and 16379 are excluded as normal Redis cluster traffic. Review `RemoteIPs` against threat intel feeds.

### Query 2 — Suspicious Child Process Spawned by redis-server
*Targets: Post-exploitation shell spawning or tool staging following Lua `os.execute()` or `io.popen()` sandbox escape.*

event_simpleName IN ("ProcessRollup2", "SyntheticProcessRollup2")
  ParentBaseFileName IN ("redis-server", "redis-sentinel")
| eval CmdLineLower=lower(CommandLine)
| where match(CmdLineLower, "(bash|sh|dash|python|perl|ruby|curl|wget|nc|ncat|nmap|chmod|base64|/tmp/|/dev/shm/|mkfifo|socat)")
| stats count, values(CommandLine) as CommandLines, values(MD5HashData) as Hashes,
        values(FileName) as ChildProcesses, earliest(ContextTimeStamp) as FirstSeen,
        latest(ContextTimeStamp) as LastSeen
  by ComputerName, UserName, ParentBaseFileName, ParentProcessId
| sort -count
> **Analyst Note:** Redis spawning any shell interpreter or network utility is a high-confidence indicator of compromise. `/tmp/` and `/dev/shm/` writes are classic staging locations for dropped payloads following Redis RCE. Escalate any hit immediately.

### Query 3 — File System Artefacts — Suspicious Writes by Redis Process
*Targets: Payload staging, cron persistence, SSH key injection, or webshell drops associated with post-exploitation.*

event_simpleName IN ("PeCreateFileOpen", "PeFileWritten", "NewExecutableRenamed")
  ContextImageFileName IN ("*redis-server*", "*redis-cli*", "*/redis/*")
| eval PathLower=lower(TargetFileName)
| where match(PathLower, "(/tmp/|/dev/shm/|/var/spool/cron|/etc/cron|/root/\.ssh/|/home/.*\.ssh/|\.php$|\.py$|\.sh$|\.elf$)")
| stats count, values(TargetFileName) as WrittenPaths, values(SHA256HashData) as Hashes,
        earliest(ContextTimeStamp) as FirstSeen
  by ComputerName, UserName, ContextImageFileName, ContextProcessId
| sort -count
> **Analyst Note:** Authorised Redis processes should not write executable files, cron entries, or SSH keys. SSH key injection (`/root/.ssh/authorized_keys`) is a well-documented persistence mechanism following Redis RCE. Hash all artefacts and submit to VirusTotal / internal sandbox.

## nfdump Netflow Queries

### Query 1 — Inbound Connections to Redis Default Port from External Sources
*Targets: Exploitation attempts against exposed Redis instances on port 6379.*

nfdump -r /var/log/netflow/nfcapd.current \
  -f 'proto tcp and dst port 6379 and not src net 10.0.0.0/8 and not src net 172.16.0.0/12 and not src net 192.168.0.0/16' \
  -o 'fmt:%ts %sa %da %sp %dp %byt %pkt %fl' \
  -s record/bytes -n 50
> **Analyst Note:** Redis port 6379 should never be reachable from external/untrusted networks. Any external source IP hitting this port is a high-priority finding. Cross-reference source IPs against CISA KEV-associated threat intel and known scanning infrastructure (e.g., Shodan-indexed scanners). Escalate non-scanner hits immediately.

### Query 2 — Anomalous Outbound Traffic Volume from Redis Hosts Following Inbound 6379 Sessions
*Targets: Data exfiltration or C2 staging traffic following a successful exploit — look for hosts that received inbound Redis traffic and then initiated anomalous outbound flows.*

nfdump -r /var/log/netflow/nfcapd.current \
  -f 'proto tcp and src net REDIS_HOST_SUBNET/24 and dst port in [80,443,4444,1337,8080,8443,9001,31337] and not dst net 10.0.0.0/8 and not dst net 172.16.0.0/12 and not dst net 192.168.0.0/16' \
  -A srcip,dstip,dstport \
  -s record/bytes -n 30 \
  -o 'fmt:%sa %da %dp %byt %pkt %fl %ts %te'
> **Analyst Note:** Replace `REDIS_HOST_SUBNET` with your Redis server network range. Outbound connections to non-standard high ports (4444, 1337, 31337) from Redis hosts are strong post-exploitation indicators. High byte counts on port 443 may indicate encrypted C2 or exfil. Correlate with CrowdStrike Query 1 results for the same source host and timestamp window.

### Query 3 — High-Frequency Short-Duration Connections to Redis Port (Exploitation Scanning / Brute Probe)
*Targets: Automated exploitation tooling or scanning waves targeting Redis instances — characterised by many short-lived TCP sessions.*

nfdump -r /var/log/netflow/nfcapd.current \
  -f 'proto tcp and dst port 6379 and flags S and duration < 2' \
  -A srcip,dstip \
  -s record/flows -n 25 \
  -o 'fmt:%sa %da %fl %pkt %byt %ts %te'
> **Analyst Note:** High flow counts with sub-2-second durations and SYN-only or SYN/RST patterns indicate automated scanning or exploit-framework probing (e.g., Metasploit redis modules, custom Lua-injection scripts). A single source IP generating >50 flows in a short window to port 6379 should be blocked and investigated. Compare source IPs against known mass-scanner lists before escalating as targeted activity.
