## CrowdStrike Event Search

### Query 1 — Outbound kubectl Connections to Non-Standard / Attacker-Controlled API Servers

The core exploit mechanism redirects `kubectl` to an attacker-controlled API server via injected `--server` flag [2][3]. This query hunts for `kubectl` processes making outbound connections to ports typically used for Kubernetes API servers (6443, 8443, 443) to IPs that are **not** your known cluster control-plane addresses.

event_simpleName=NetworkConnectIP4
  ImageFileName IN ("*kubectl*", "*kubectl.exe*")
| where RemotePort IN (443, 6443, 8443, 8080)
| where RemoteIP NOT IN ("<your-legitimate-apiserver-ip-1>", "<your-legitimate-apiserver-ip-2>")
| stats count, values(RemoteIP) as RemoteIPs, values(RemotePort) as Ports,
         dc(RemoteIP) as UniqueIPs
  by ComputerName, UserName, ImageFileName, CommandLine
| where UniqueIPs > 0
| sort -count

> **Analyst note:** Replace the `NOT IN` list with your authorised Kubernetes API server IPs. Any `kubectl` process connecting outbound to an IP not in that allowlist while this CVE is unpatched is high-priority.

### Query 2 — MCP Server Kubernetes Process Spawning kubectl with --server Flag

The argument injection requires `resourceType` or `name` parameters to inject `--server` [3][4]. Hunt for the MCP server process spawning `kubectl` where `--server` appears in the command line pointing to non-internal addresses.

event_simpleName=ProcessRollup2
  ParentBaseFileName IN ("node", "node.exe", "mcp-server-kubernetes", "mcp-server-kubernetes.exe")
  BaseFileName IN ("kubectl", "kubectl.exe")
  CommandLine="*--server*"
| rex field=CommandLine "--server[= ](?P<injected_server>[^\s]+)"
| where NOT match(injected_server, "^https?://(10\.|172\.(1[6-9]|2[0-9]|3[01])\.|192\.168\.)")
| stats count, values(CommandLine) as CommandLines, values(injected_server) as InjectedServers
  by ComputerName, UserName, ParentBaseFileName, BaseFileName
| sort -count

> **Analyst note:** This directly targets the injection vector described in [2][3] — `--server` pointing to a non-RFC-1918 address from the MCP parent process is the primary detection signal.

### Query 3 — Post-Exploitation: kubectl Credential / Token File Access Following Anomalous API Server Connection

If bearer token leakage occurs, adversaries may attempt to persist or exfiltrate kubeconfig / service-account tokens. Hunt for file reads of token paths shortly after anomalous kubectl network activity.

event_simpleName IN ("GenericFileNameDmrRecord", "FileOpenInfo")
  FileName IN (
    "*/.kube/config",
    "*/var/run/secrets/kubernetes.io/serviceaccount/token",
    "*/run/secrets/kubernetes.io/serviceaccount/token",
    "*kubeconfig*",
    "*/serviceaccount/token"
  )
| join type=inner
  [search event_simpleName=NetworkConnectIP4
     ImageFileName IN ("*kubectl*")
     RemotePort IN (443, 6443, 8443, 8080)
     RemoteIP NOT IN ("<your-legitimate-apiserver-ip-1>", "<your-legitimate-apiserver-ip-2>")
   | fields ComputerName, UserName, ContextProcessId as TargetProcessId]
  by ComputerName, UserName
| stats count, values(FileName) as AccessedFiles, values(ContextTimeStamp) as Timestamps
  by ComputerName, UserName
| sort -count

> **Analyst note:** Token path list covers both developer workstation kubeconfigs and in-cluster service-account injection paths. Correlating file access with the anomalous network event tightens precision.

## nfdump Netflow Queries

### Query 1 — Outbound Flows from kubectl/MCP Hosts to Non-Cluster API Server IPs on Kubernetes Ports

nfdump -r /var/log/netflow/nfcapd.current \
  -f 'proto tcp and (dst port 6443 or dst port 8443 or dst port 443) \
      and not dst net 10.0.0.0/8 \
      and not dst net 172.16.0.0/12 \
      and not dst net 192.168.0.0/16 \
      and not dst ip <legitimate-apiserver-ip-1> \
      and not dst ip <legitimate-apiserver-ip-2> \
      and src ip <mcp-server-host-subnet>' \
  -s record/bytes -n 50 \
  -o extended

> **Analyst note:** Scope `src ip` to subnets where your MCP Server Kubernetes deployments run. Flows exiting to external IPs on 6443/8443 from these hosts are anomalous and consistent with `--server` injection [2][3].

### Query 2 — Anomalous Outbound Volume from MCP Hosts (Bearer Token Exfiltration Indicator)

A successful token exfiltration would likely produce a short-lived but notably large outbound flow immediately after the injected kubectl call. Hunt for high-byte-count short-duration TCP flows.

nfdump -r /var/log/netflow/nfcapd.current \
  -f 'proto tcp and (dst port 443 or dst port 6443 or dst port 8443) \
      and src ip <mcp-server-host-subnet> \
      and bytes > 50000 and duration < 5' \
  -s record/bytes -n 30 \
  -o "fmt:%ts %td %sa %da %dp %sp %byt %pkt %fl"

> **Analyst note:** Tune `bytes > 50000` and `duration < 5` to your baseline. Bearer token + kubeconfig exfiltration is small in payload but may appear as a fast burst. Lower the byte threshold if your baseline supports it.
