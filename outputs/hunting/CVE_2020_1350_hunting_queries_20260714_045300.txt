## CrowdStrike Event Search

### Query 1 — Anomalous DNS Server Process Spawning Children (Post-Exploitation)
SIGRed targets `dns.exe` (Windows DNS Server service). Successful exploitation runs code as SYSTEM under the DNS server process. Hunt for unusual child processes.

event_simpleName IN ("ProcessRollup2", "SyntheticProcessRollup2")
| search ParentBaseFileName="dns.exe"
| search FileName NOT IN ("dns.exe", "conhost.exe")
| eval suspicion=case(
    FileName IN ("cmd.exe","powershell.exe","wscript.exe","cscript.exe","mshta.exe","rundll32.exe","regsvr32.exe"), "HIGH",
    FileName IN ("net.exe","net1.exe","whoami.exe","nltest.exe","ipconfig.exe"), "MEDIUM",
    true(), "LOW"
  )
| stats count by ComputerName, UserName, ParentBaseFileName, FileName, CommandLine, suspicion
| where suspicion IN ("HIGH","MEDIUM")
| sort -suspicion, -count

### Query 2 — DNS Server Network Connections to Unexpected External Destinations
DNS servers should not initiate outbound connections to arbitrary external IPs on non-DNS ports. Post-exploitation callbacks from `dns.exe` are a strong indicator of compromise.

event_simpleName="NetworkConnect"
| search ImageFileName="*\\dns.exe"
| eval RemotePort_int=tonumber(RemotePort)
| where RemotePort_int NOT IN (53, 5353, 5355)
| eval direction=if(isnotnull(RemoteIP), "outbound", "unknown")
| eval rfc1918=if(
    match(RemoteIP, "^10\\.") OR
    match(RemoteIP, "^172\\.(1[6-9]|2[0-9]|3[01])\\.") OR
    match(RemoteIP, "^192\\.168\\."),
    "internal", "external"
  )
| where rfc1918="external"
| stats count, earliest(_time) as first_seen, latest(_time) as last_seen
    by ComputerName, ImageFileName, RemoteIP, RemotePort
| sort -count

### Query 3 — Registry and File Artefacts: DNS Server Service Tampering
Post-exploitation may involve persistence via registry modification or dropping payloads in DNS-related paths.

(event_simpleName IN ("RegGenericValueSet","RegSetValue")
    RegistryPath="*\\services\\DNS*")
OR
(event_simpleName IN ("PeFileWritten","NewExecutableWritten")
    FilePath IN ("*\\System32\\dns*","*\\SysWOW64\\dns*","*\\Windows\\Temp\\*"))
| eval artefact_type=case(
    event_simpleName IN ("RegGenericValueSet","RegSetValue"), "REGISTRY_MOD",
    event_simpleName IN ("PeFileWritten","NewExecutableWritten"), "BINARY_DROP",
    true(), "OTHER"
  )
| stats count, earliest(_time) as first_seen, values(FilePath) as file_paths,
    values(RegistryPath) as reg_paths
    by ComputerName, UserName, artefact_type, event_simpleName
| sort -count

## nfdump Netflow Queries

### Query 1 — Oversized DNS TCP Traffic Indicative of SIGRed Exploit Attempt
SIGRed is triggered via a crafted DNS-over-TCP response with an oversized SIG record (>65,535 bytes). Hunt for anomalously large DNS TCP flows — normal DNS TCP messages are rarely >512 bytes for queries; legitimate AXFR transfers are the main exception.

nfdump -r /var/log/netflow/nfcapd.current \
  -f 'proto tcp and port 53' \
  -A srcip,dstip,srcport,dstport \
  -o 'fmt:%ts %te %sa %da %sp %dp %pkt %byt %fl' \
  -n 50 \
  -s record/bytes \
  | awk '$8 > 65535 {print $0}'

> **Analyst Note:** Filter to flows where bytes > 65535 on TCP/53. Legitimate zone transfer (AXFR) traffic may also appear here — correlate destination IP against known secondary DNS servers to reduce false positives.

### Query 2 — Unexpected Outbound Connections FROM DNS Server Hosts Post-Exploit
Hunt for outbound C2 beacon traffic originating from hosts running Windows DNS Server (identified by internal asset inventory). Common C2 ports targeted post-SIGRed compromise.

nfdump -r /var/log/netflow/nfcapd.current \
  -f 'proto tcp and src ip x.x.x.x and dst port in [80,443,4444,8080,8443] and not dst net 10.0.0.0/8 and not dst net 172.16.0.0/12 and not dst net 192.168.0.0/16' \
  -s record/bytes \
  -n 25 \
  -o 'fmt:%ts %te %sa %da %sp %dp %pkt %byt %pr %flg'

> **Analyst Note:** Replace `x.x.x.x` with the IP(s) of your Windows DNS Server hosts. DNS servers have no legitimate reason to initiate outbound HTTP/HTTPS to arbitrary external IPs. Any hits here warrant immediate triage.

---

### Query 3 — High-Volume DNS TCP Connection Spike (Reconnaissance or Exploit Loop)
Hunt for abnormal TCP/53 connection counts from a single external source to DNS servers — may indicate scanning, exploit attempts, or exploit retries.

nfdump -r /var/log/netflow/nfcapd.current \
  -f 'proto tcp and dst port 53 and dst ip x.x.x.x' \
  -A srcip \
  -s record/flows \
  -n 30 \
  -o 'fmt:%sa %fl %pkt %byt' \
  | sort -k2 -rn \
  | head -20

> **Analyst Note:** Replace `x.x.x.x` with your DNS server IP(s). Baseline normal TCP/53 flow counts in your environment first. Internet-facing DNS resolvers will have higher legitimate counts — focus on authoritative servers that should not receive recursive queries from arbitrary external IPs.

## Hunting Notes

### What to Look For

| Signal | Significance |
|---|---|
| `dns.exe` spawning shells (`cmd.exe`, `powershell.exe`) | **Critical** — direct evidence of RCE under DNS service context |
| `dns.exe` initiating non-DNS outbound TCP | **High** — likely C2 callback post-exploitation |
| DNS TCP flows > 65,535 bytes to your DNS servers | **High** — matches SIGRed exploit payload size requirement |
| New executables written to `%SystemRoot%\System32\` by DNS service | **High** — persistence mechanism post-RCE |
| Registry modifications under `HKLM\SYSTEM\CurrentControlSet\Services\DNS` | **Medium** — could indicate service configuration tampering for persistence |
| Lateral movement from DNS server host (SMB, WMI, RDP) | **High** — DNS servers run as SYSTEM, making them prime pivot points |

### False Positive Considerations

- **DNS TCP traffic:** Legitimate AXFR zone transfers between authoritative servers will appear in Query 1 (nfdump) and Query 2 (CrowdStrike). Maintain a whitelist of known secondary DNS server IPs.
- **`dns.exe` child processes:** Some monitoring and backup agents legitimately inject into or are spawned from service contexts. Validate against your approved software inventory.
