# Threat Hunting Queries — CVE-2026-60002
**OpenSSH Use-After-Free (Host Key Re-Exchange) | CVSS 7.7 HIGH [1] | RCE: Network-Exploitable (AV:N/PR:N/UI:N)**
**Priority Tier: CRITICAL — ACT NOW | Tags: [RCE] [HIGH] [T1] [WIDE]**

OpenSSH versions prior to 10.4 [2][3] are affected by a use-after-free triggered when a server changes its host key during SSH key re-exchange. A client-side attacker can cause memory corruption on the connecting client; exploitation requires the server to perform a host key change mid-session [4]. No public proof-of-concept is currently known.

---

## CrowdStrike Event Search

### Query 1 — Inbound SSH Network Connections to Vulnerable OpenSSH Hosts (Port 22)
Targets hosts accepting SSH connections that may be running pre-10.4 OpenSSH, flagging connections from unusual or external source IPs for analyst review.

event_simpleName=NetworkConnectIP4
| search LocalPort=22
| eval IsExternal=if(match(RemoteIP,"^10\.|^172\.(1[6-9]|2[0-9]|3[01])\.|^192\.168\."),0,1)
| where IsExternal=1
| stats count dc(RemoteIP) AS UniqueSourceIPs
    by ComputerName, LocalPort, aid
| where count > 5
| sort -count
*Hunting focus:* External hosts initiating multiple SSH sessions to a single endpoint — potential scanning or repeated exploitation attempts. Adjust threshold and internal IP ranges to match your environment.

---

### Query 2 — Post-Exploitation Process Spawning from SSHD
Targets anomalous child processes spawned by `sshd` that are inconsistent with legitimate SSH session activity, which may indicate successful memory corruption / RCE following exploitation [3].

event_simpleName=ProcessRollup2
| search ParentBaseFileName IN ("sshd", "sshd-session")
| where FileName NOT IN (
    "bash", "sh", "zsh", "sftp-server", "scp",
    "rsync", "git", "sudo", "env", "id"
  )
| stats count values(CommandLine) AS Commands
    by ComputerName, UserName, FileName, ParentBaseFileName, aid
| sort -count
*Hunting focus:* Unexpected binaries (`curl`, `wget`, `python3`, `perl`, `nc`, `chmod`, shell interpreters with unusual arguments) launched directly under `sshd`. High-confidence escalation trigger. Tune the exclusion list to your baseline SSH usage.

---

### Query 3 — SSH Host Key File Modification (Server-Side Indicator)
The exploit requires the server to change its host key during re-exchange [4]. Detect unexpected writes to SSH host key files, which may indicate attacker-controlled key rotation or staging.

event_simpleName=AsynchronousImageLoad OR event_simpleName=ModifyFile
| search TargetFileName IN (
    "/etc/ssh/ssh_host_rsa_key",
    "/etc/ssh/ssh_host_ed25519_key",
    "/etc/ssh/ssh_host_ecdsa_key",
    "/etc/ssh/ssh_host_dsa_key",
    "/etc/ssh/ssh_host_rsa_key.pub",
    "/etc/ssh/ssh_host_ed25519_key.pub"
  )
| where FileName NOT IN ("sshd", "ssh-keygen", "ansible", "puppet", "chef-client")
| stats count values(FileName) AS WritingProcess values(UserName) AS Users
    by ComputerName, TargetFileName, aid
| sort -count
*Hunting focus:* Host key files should rarely change outside planned rotation. Any modification by non-authorised processes is a high-priority escalation indicator given the exploit mechanism described [2][4].

## nfdump Netflow Queries

### Query 1 — High-Frequency SSH Connection Attempts to Internal Hosts (Scanning / Exploitation Attempts)
nfdump -r /var/log/netflow/nfcapd.current \
  -f 'proto tcp and dst port 22' \
  -A srcip,dstip \
  -s record/flows \
  -n 50 \
  -o "fmt:%sa -> %da | flows: %fl | pkts: %pkt | bytes: %byt"
*Hunting focus:* Source IPs generating disproportionate flow counts toward port 22 across multiple destination addresses may indicate reconnaissance or bulk exploitation attempts. Investigate sources with flows >> your SSH baseline.

---

### Query 2 — Short-Lived SSH Sessions (Potential Failed Exploit or Crash-and-Reconnect Pattern)
Use-after-free exploitation may cause client or server crashes, resulting in many very short-duration TCP sessions on port 22.

nfdump -r /var/log/netflow/nfcapd.current \
  -f 'proto tcp and dst port 22 and duration < 5' \
  -A srcip,dstip \
  -s record/flows \
  -n 30 \
  -o "fmt:%sa -> %da | dur: %td | flows: %fl | pkts: %pkt | flags: %flg"
*Hunting focus:* Legitimate SSH sessions rarely complete in under 5 seconds. A cluster of sub-5-second TCP flows from the same source IP to port 22, especially with RST flags, is consistent with repeated crash-triggered reconnects or fuzzing behaviour.

---

### Query 3 — Anomalous Outbound Traffic Following Inbound SSH Session (Post-RCE C2 Beaconing)
If exploitation succeeds and grants execution, expect outbound C2 or data exfiltration traffic shortly after an inbound SSH connection window.

nfdump -r /var/log/netflow/nfcapd.current \
  -f 'proto tcp and src port != 22 and dst port in [443,80,4444,8080,8443,1337] and dst ip not x.x.x.x/8' \
  -A srcip,dstip,dstport \
  -s record/bytes \
  -n 25 \
  -o "fmt:%sa:%sp -> %da:%dp | bytes: %byt | pkts: %pkt | start: %ts"
*Hunting focus:* Correlate the source IPs here with IPs observed receiving inbound SSH within the same time window. Unexpected outbound HTTPS or high-port traffic from SSH-exposed hosts post-session is a strong escalation indicator. Replace `x.x.x.x/8` with your internal RFC1918 ranges to exclude legitimate internal traffic.

## Hunting Notes

### What to Look For

| Signal | Relevance |
|---|---|
| OpenSSH process version strings < 10.4 on internet-facing hosts | Direct exposure to CVE-2026-60002 [2] |
| SSH host key files modified outside planned rotation windows | Core exploit precondition — server-side host key change [4] |
| `sshd` spawning unusual child processes | Post-RCE execution artefact [3] |
| Repeated short-duration TCP/22 sessions from a single source | Crash-loop pattern consistent with exploit probing |
| Outbound beaconing traffic shortly after inbound SSH session | Post-exploitation C2 initiation |

### False Positive Considerations
- **Configuration management tools** (Ansible, Puppet, Chef, Terraform) legitimately modify SSH host key files during provisioning — build an allowlist of their process names and invocation accounts.
- **Legitimate SCP/SFTP sessions** may spawn short-lived child processes under `sshd`; ensure `sftp-server` and `scp` are in exclusion lists (Query 2).
- **Load balancers and SSH jump hosts** will show high inbound SSH flow counts by design — establish per-host
